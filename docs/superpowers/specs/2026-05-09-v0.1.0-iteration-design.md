**Date**: 2026-05-09

## Phase 2: TUI Interaction Improvements

### 2.1 Model Tier Mapping Display

**Problem**: Users don't know what haiku/sonnet/opus/subagent model they'll get when selecting a model.

**Solution**: In `showModels()`, add a `tview.TextView` tier info bar above the help text. On model highlight change, compute and display:

```
haiku: qwen2.5:7b | sonnet: qwen3-coder | opus: qwen2.5:32b | sub: qwen2.5-coder:7b
```

For `ForceModelTiers` providers:

```
all tiers: qwen3-coder:480b
```

**Implementation**: Add `tierInfo *tview.TextView` to `tuiState`. In `showModels()`, set `modelList.SetChangedFunc` to update tier info with `withSelectedModel(preset, selectedModel)`. Compute tier labels from the resulting preset's Haiku/Sonnet/Opus/Subagent fields.

### 2.2 Quick Switch in Provider Details

**Problem**: Must enter model page to switch; unnecessary for users who want the default model.

**Solution**: Add "Switch (default)" action in `showDetail()`:

- Shortcut key: `s`
- Behavior: Calls `finishSelection(provider, preset.Model)` using the provider's default model
- Only shown when API key is available

### 2.3 Provider Links in Details

**Problem**: No easy way to visit provider website or get an API key from the TUI.

**Solution**: In `showDetail()`, show Website and APIKeyURL from preset when available:

```
Website:  https://openrouter.ai
Get Key:  https://openrouter.ai/keys
```

### 2.4 Terminal Output Beautification

**Problem**: CLI output (current, switch, list, etc.) is plain text without color or alignment.

**Solution**: Add color-coded output for CLI commands:

- `cs current`: label dim/gray, value bright/white, provider name green
- `cs switch`: success prefix `[OK]` in green, error prefix `[ERR]` in red
- `cs list`/`list --verbose`: aligned columns, `saved`/`not saved` replaced with ✓/✗
- Detect `NO_COLOR` env var and `TERM=dumb` to disable colors

Implementation: Add `color.go` with helper functions `green()`, `red()`, `dim()`, `bold()` that wrap text with ANSI codes when colors are enabled.

## Phase 3: DeepSeek Codex Provider (Conditional)

### Prerequisite

Verify that DeepSeek provides a Responses API compatible endpoint (`/v1/responses`). This can be tested with:

```bash
curl -X POST https://api.deepseek.com/v1/responses \
  -H "Authorization: Bearer $DEEPSEEK_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"model":"deepseek-v4-pro","input":"Say hi"}'
```

### If Compatible

Add Codex DeepSeek provider:

1. Add `codexDeepSeekPreset()` in `agent.go`:
   - `BaseURL`: `"https://api.deepseek.com/v1"`
   - `AuthEnv`: `"DEEPSEEK_API_KEY"`
   - `ForceModelTiers`: `true`
   - Default model: `"deepseek-v4-pro"`

2. Update `resolveAgentProviderPreset` and `resolveAgentSwitchPreset` for `case "deepseek"`

3. Add `"deepseek"` to `providerNamesForAgent` Codex branch

4. Update `codexTOMLProviderName` / `codexTOMLProviderKey` mapping

5. Update Codex whitelist in `cmdSetKey`

6. Add comprehensive tests for Codex DeepSeek switch, env, current, TOML generation

### If Not Compatible

Do not expose to users. Optionally add a commented placeholder in the code for future implementation when DeepSeek adds Responses API support.
